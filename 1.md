# 代码编写规范

  - 以下任何一个环节，如果skill可用，调用skills
---

## 工作流程

### 1. 理解需求

仔细分析用户输入：
  - 用户希望我完成什么任务
  - 任务目标和约束条件是什么
  - 如果用户只是询问技术问题，则不需要编写代码。

**相关规范：**

- **明确已知条件、限制、目标**：任何不明确的条件应当询问用户，禁止假设和猜测
  > 建议：列出「已知/未知清单」，未知项逐一询问确认

- **用户询问方案时，比较各方案优缺点并给出建议**
  > 建议：使用表格对比各方案的优缺点、适用场景、风险点

- **围绕用户的需求以专业、友好的风格沟通**

---

### 2. 规划方案

思考最佳实现方案，选择合适的技术栈和设计模式。
  - 完成这个任务有哪些方案？
  - 比较各方案的优缺点、适用场景、风险点
  - 向用户报告编写方案，请用户确认
  - 确认方案，根据方案列出步骤

**相关规范：**

- **用户发布任务时，列出任务步骤，拆分任务为任务树**
  > 建议：使用 TodoWrite 工具创建任务列表，按优先级排序

- **编写/修改代码应当明确接口有无变化、会不会影响下游**
  > 检查清单：
  > - 函数签名是否变更？
  > - 返回值类型是否变更？
  > - 是否新增/删除了公开 API？
  > - 调用方是否需要同步修改？

- **所有细节、变更、影响应当通知用户防止偏离预期**

- **搜索网络以确保掌握最新的库动态**
  > 常用搜索关键词：`库名 + latest version`、`库名 + best practices 2026`、`库名 + migration guide`

---

### 3. 实现方案

编写干净、注释清晰的代码，积极进行防御性编程、合理运用语言高级特性。

**相关规范：**

- **优先使用 uv（uv add、uv remove、uv run）/npm，除非用户要求**
  > 常用命令：
  > - Python：`uv add <pkg>`、`uv remove <pkg>`、`uv run pytest`
  > - Node.js：`npm install <pkg>`、`npm run build`

- **凡是 uv、gradle 等需要拉取包的情况，统一使用镜像源**
  > 超时无进展的一律中止，不要反复尝试

- **复杂任务均使用 sub agent 逐步完成**

- **代码必须符合常见的语言和框架编码规范**
  > 参考：
  > - Python：PEP 8、Google Python Style Guide
  > - TypeScript/React：Airbnb Style Guide、Vercel React Best Practices
  > - Rust：Rust API Guidelines

- **代码行为必须严格符合预期，否则应当暂停并通知用户问题**

- **必须包含必要的注释说明**
  > 注释类型：
  > - 文件级：模块用途、对外暴露的函数/类
  > - 函数级：参数说明、返回值说明、复杂逻辑解释
  > - 行内：仅用于解释「为什么」，而非「是什么」

- **必须考虑错误处理和边界情况**
  > 常见边界情况：
  > - 空输入/空列表
  > - 极大/极小值
  > - 特殊字符/非法输入
  > - 并发访问/竞态条件
  > - 网络超时/服务不可用

- **禁止使用弃用的功能**

---

### 4. 自我检查

编写代码之后必须通过静态分析、单元测试、集成测试、动态调试等方式验证代码逻辑。

**测试要求：**

- **前端使用 Vitest + React Testing Library 进行集成测试和单元测试，使用 Playwright 进行 E2E 测试**
  > 重点：组件响应是否符合预期、关键 hooks 和函数的行为是否符合预期

- **后端使用 pytest 编写单元测试和集成测试**
  > 每个模块的对外接口要100%测试覆盖，要对边界情况和错误情况进行测试

- **修改模块之后，如果修改影响其功能，运行该模块的测试**

**相关规范：**

- **编写完代码要进行检查**：这次的回答逻辑是否贯通、是否是最优化的方案、是否符合用户的预期
  > 检查清单：
  > - 是否满足所有用户要求？
  > - 是否有遗漏的边界情况？
  > - 是否有更简洁的实现方式？
  > - 是否引入了不必要的复杂性？

- **排查故障应当使用调试工具**

- **需要用户协助的时候随时指出**
